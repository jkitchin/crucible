* System Overview

Crucible consists of four layers:

1. *File system*: org-mode wiki articles and primary source files
2. *Database*: SQLite graph database tracking relationships
3. *Embeddings*: vector representations for semantic search
4. *CLI*: Python click commands for ingestion, search, and maintenance

The LLM (Claude Code via the crucible skill) sits above all four layers,
reading and writing files, querying the database via CLI commands, and
maintaining the knowledge base.

* Database Schema

The wiki graph database (=db/crucible.db=) models six entity types and
their relationships:

** Entity Tables

*** sources

Primary source documents (papers, notebooks, data files).

| Column | Type | Description |
|-----------+---------+----------------------------------------|
| id | INTEGER | Primary key |
| path | TEXT | Relative path in sources/ |
| title | TEXT | Source title |
| source_type | TEXT | pdf, web, notebook, data, other |
| shareable | INTEGER | 0 = external/gitignored, 1 = shareable |
| url | TEXT | Original URL (optional) |
| authors | TEXT | JSON array |
| date | TEXT | ISO date |
| ingested_at | TEXT | Timestamp |
| metadata | TEXT | JSON blob for extras |

*** articles

Distilled wiki articles (the org files in =wiki/=).

| Column | Type | Description |
|---------------+---------+-------------------------------|
| id | INTEGER | Primary key |
| path | TEXT | Relative path in wiki/ |
| title | TEXT | Article title |
| article_type | TEXT | concept, summary, comparison, method |
| status | TEXT | draft, reviewed, verified |
| created_at | TEXT | Timestamp |
| updated_at | TEXT | Timestamp |
| distill_model | TEXT | Which LLM produced it |
| abstract | TEXT | Brief summary |

*** concepts

Named topics that articles can be tagged with, extracted from =#+FILETAGS=.

| Column | Type | Description |
|---------+---------+------------------------------|
| id | INTEGER | Primary key |
| name | TEXT | Canonical name (unique) |
| aliases | TEXT | JSON array of alternate names |

** Relationship Tables

*** article_sources

Many-to-many: which sources an article was distilled from.

*** article_concepts

Many-to-many: which concepts an article covers.

*** article_links

Directed graph: explicit cross-references between articles, extracted
from =[[file:...]]= links in the org files.

*** article_derivations

Tracks knowledge evolution: article Y was synthesized from articles
A, B, C at a given time. Extracted from the =:DERIVED_FROM:= property.

| Column | Type | Description |
|------------+---------+------------------------------|
| derived_id | INTEGER | The synthesized article |
| source_id | INTEGER | One of its input articles |
| derived_at | TEXT | When the derivation happened |

*** peers

Registered peer crucible instances for cross-querying.

| Column | Type | Description |
|----------+------+---------------------------|
| name | TEXT | Short alias (primary key) |
| path | TEXT | Absolute path to peer root |
| added_at | TEXT | When registered |

** Full-Text Search

The =article_fts= virtual table uses SQLite FTS5 with porter stemming
tokenization. It indexes article path, title, and full text content.
Populated by =crucible sync=.

** Embeddings

The =embeddings= table stores vector representations generated by ollama.

| Column | Type | Description |
|------------+---------+--------------------------------|
| article_id | INTEGER | References articles(id) |
| model | TEXT | Embedding model name |
| dimensions | INTEGER | Vector dimensionality (e.g. 768) |
| vector | BLOB | Packed float32 array |
| updated_at | TEXT | Timestamp |

* Search Architecture

** FTS5 Keyword Search

Standard SQLite full-text search with porter stemming. The query is
matched against article path, title, and content. Results are ranked
by BM25 (SQLite's default FTS5 ranking).

** Semantic Vector Search

1. The query is embedded using the same ollama model as the articles
2. All article embeddings are loaded and cosine similarity is computed
3. Results are sorted by similarity score

This is a brute-force scan, appropriate for knowledge bases up to a
few thousand articles. For larger scale, the embeddings table could
be migrated to use libSQL's vector index.

** Hybrid Search with Reciprocal Rank Fusion

Both FTS and semantic search produce ranked lists. These are merged
using Reciprocal Rank Fusion (RRF):

For each article appearing in either list:
=score = fts_weight / (k + fts_rank) + semantic_weight / (k + semantic_rank)=

Where =k = 60= (the standard RRF constant that dampens rank differences).
Articles found by both methods get contributions from both, so they
naturally rank higher.

The weights are configurable via =--fts-weight= and =--semantic-weight=.

** Cross-Peer Search

FTS5 virtual tables cannot be queried across SQLite ATTACH boundaries
with the MATCH operator. Instead, cross-peer search opens a separate
SQLite connection to each peer's database and runs the query locally.
Results from all connections are merged and re-ranked.

* Org-Mode Parser

The lightweight parser in =src/crucible/orgparse.py= extracts metadata
from org files without requiring Emacs. It handles:

- =#+TITLE:= keyword
- =#+FILETAGS:= keyword (colon-delimited tags)
- =:PROPERTIES:= drawers (key-value pairs within drawer boundaries)
- =[[file:...]]= links (with optional =::= anchors)
- =cite:=, =citep:=, =citet:= org-ref citations

It is not a full org-mode parser. Complex structures (nested drawers,
inline properties, dynamic blocks) are not supported but are not needed
for crucible's metadata extraction.

* Sync Process

=crucible sync= operates in two passes to avoid foreign key violations:

1. *Pass 1*: scan all org files, register/update articles in the database,
   create concepts from filetags, and link articles to concepts.
2. *Pass 2*: process inter-article links (=[[file:...]]=), source links
   (=:SOURCE_KEYS:=), and derivation chains (=:DERIVED_FROM:=).

The two-pass approach ensures all articles exist before cross-referencing.

* Peer Architecture

Peers are registered in the =peers= table with a name and absolute path.
When =--all= is used on a query, crucible opens separate read-only
connections to each peer's =db/crucible.db= and runs the query against it.
Results are tagged with the peer name and merged with local results.

No data is written to peer databases. The connection is read-only by
convention (SQLite's default mode allows writes, but crucible never
issues write queries against peer connections).
